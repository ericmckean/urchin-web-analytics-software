Urchin 6.6, 6.601 및 6.602 자주 발생하는 문제

Urchin 6.602 - 개정: 2009년 10월 23일

관리

  • 목표 구성 마법사에서 사용하는 사이트 오버레이 기술은 비밀번호로 차단되는 페이지는 처리하지 못하며 프레임, javascript, css, 자동 리디렉션 등이 포함된 복잡한 페이지는 추적하는 데 문제가 있습니다.

지리 데이터

  • 새 지리 데이터 데이터베이스의 형식이 달라 로컬 IP 주소와 도메인 이름 간의 매핑 정보를 가져올 수 없습니다.

로그 처리

  • 트래픽이 많이 발생하는 사이트에 대해 시간 순서로 로그를 처리하려면 속도가 크게 저하될 수 있습니다.
  • 특히 트래픽이 많이 발생하는 사이트에서 기본 100,000 레코드 이상으로 DB 한도를 확대하면 로그 처리 성능이 저하될 수 있습니다.
  • 로그 처리 엔진은 처리 시 사용된 각 로그 파일에 대해 프로필에 적용된 필터별로 메모리를 할당합니다. 따라서 필터가 많이 구성된 프로필에서 과도한 와일드카드 확장검색을 사용하는 등의 이유로 다수의 로그 파일이 로그 소스와 일치할 경우 대량의 메모리 할당 요청이 발생하여 필요한 메모리가 할당되지 못할 수 있습니다.
  • 특히 로그 처리 시 다수의 로그 파일을 사용할 경우 조회 테이블 크기가 증가함에 따라 메모리 요구량도 함께 증가하여 결과적으로 메모리를 할당하지 못할 수 있습니다.

  • UNC 소켓을 사용할 때 자주 발생하는 문제가 몇 가지 있습니다. 자세한 내용은 UNC 로그 소스 작업을 참조하십시오.

스케줄러

  • Urchin 스케줄러, 프로필 및 CPC는 데이터베이스를 사용하여 데이터를 저장하며 시간기록을 조회하여 여러 작업을 예약합니다. 즉, 일단 데이터베이스를 구성한 후에는 해당 시간대를 변경해서는 안 됩니다. 따라서 urchin db 또는 Urchin 스케줄러가 실행되는 시스템의 시간대를 변경해서는 안 됩니다.

전자상거래

  • Urchin은 UTM javascript 호출을 통해 거래가 기록될 때 웹서버 로그에 직접 삽입되는 전자상거래 데이터나 장바구니에 의해 생성되는 명시적인 ELF2 전자상거래 로그에 기록되는 전자상거래 데이터를 모두 참조할 수 있습니다. 따라서 동일한 전자상거래 데이터가 웹서버 로그와 ELF2 로그 둘 다에 존재할 경우 Urchin에서 중복 거래를 생성하게 되므로, 웹서버 로그에 동일한 데이터가 존재하면 ELF2 로그는 처리되지 않도록 하는 것이 중요합니다.

유틸리티

  • udb-sanitizer에서 대용량 데이터가 포함된 프로필을 처리할 때 "즉시 실행" 기능의 처리 속도가 크게 저하될 수 있습니다.

보고

  • 대규모 트래픽이 발생하거나 기간이 긴 사이트에 대해 세그먼트별로 비교하는 보고서를 작성할 때 처리 속도가 크게 저하되거나 시간제한에 걸려 보고서가 생성되지 않을 수 있습니다.
  • "P/방문" 결과는 "매체 전환" 보고서와 "분석 옵션" 아래에 있는 각 하위 보고서 간에 다를 수 있습니다.

AdWords와 통합

  • Urchin 6.6+가 64-bit 버전의 Windows 2008에 설치된 경우에는 Urchin에서 AdWords에 직접 액세스할 수 없습니다.

PostgreSQL 설치

  • PostgreSQL 서버를 Linux Kernel 2.6 및 Windows에 설치하는 동안 설치 프로그램에는 시스템 사용자 "postgres"를 만들기 위한 root/admin 권한이 필요합니다.
  • Linux에서는 PostgreSQL 설치 프로그램이 glibc 2.3.3 이상을 필요로 합니다.
  • Urchin을 Linux Kernel 2.4에 설치하는 동안 함께 제공되는 PostgreSQL 서버를 선택한 경우에는 CPC 데이터 가져오기가 작동하지 않을 수 있습니다. 이 경우에는 외부 PostgreSQL 서버를 사용하는 것이 좋습니다.

Urchin 6.601 - 개정: 2009년 7월 23일

관리

  • 데이터센터 모드에서는 계정 관리자에게 Urchin 5의 제휴 관리자 권한보다 더 제한적인 한 가지 권한만 부여됩니다.
  • 목표 구성 마법사에서 사용하는 사이트 오버레이 기술은 비밀번호로 차단되는 페이지는 처리하지 못하며 프레임, javascript, css, 자동 리디렉션 등이 포함된 복잡한 페이지는 추적하는 데 문제가 있습니다.

지리 데이터

  • 새 지리 데이터 데이터베이스의 형식이 달라 로컬 IP 주소와 도메인 이름 간의 매핑 정보를 가져올 수 없습니다.

로그 처리

  • 성능상의 이유로 로그 처리 엔진은 처리 시 전체 지리 데이터 데이터베이스를 메모리에 로드해야 하는데 이로 인해 아주 작은 사이트에 대한 프로필을 처리할 때도 최소 800MB의 메모리 공간이 필요합니다. 이 때문에 로그 처리 초기화 시간도 길어져 프로필당 보통 2~5초가 더 소요됩니다.
  • 트래픽이 많이 발생하는 사이트에 대해 시간 순서로 로그를 처리하려면 속도가 크게 저하될 수 있습니다.
  • 특히 트래픽이 많이 발생하는 사이트에서 기본 100,000 레코드 이상으로 DB 한도를 확대하면 로그 처리 성능이 저하될 수 있습니다.
  • 로그 처리 엔진은 처리 시 사용된 각 로그 파일에 대해 프로필에 적용된 필터별로 메모리를 할당합니다. 따라서 필터가 많이 구성된 프로필에서 과도한 와일드카드 확장검색을 사용하는 등의 이유로 다수의 로그 파일이 로그 소스와 일치할 경우 대량의 메모리 할당 요청이 발생하여 필요한 메모리가 할당되지 못할 수 있습니다.
  • 특히 로그 처리 시 다수의 로그 파일을 사용할 경우 조회 테이블 크기가 증가함에 따라 메모리 요구량도 함께 증가하여 결과적으로 메모리를 할당하지 못할 수 있습니다.

스케줄러

  • Urchin 스케줄러, 프로필 및 CPC는 데이터베이스를 사용하여 데이터를 저장하며 시간기록을 조회하여 여러 작업을 예약합니다. 즉, 일단 데이터베이스를 구성한 후에는 해당 시간대를 변경해서는 안 됩니다. 따라서 urchin db 또는 urchin 스케줄러가 실행되는 시스템의 시간대를 변경해서는 안 됩니다.

전자상거래

  • Urchin은 UTM javascript 호출을 통해 거래가 기록될 때 웹서버 로그에 직접 삽입되는 전자상거래 데이터나 장바구니에 의해 생성되는 명시적인 ELF2 전자상거래 로그에 기록되는 전자상거래 데이터를 모두 참조할 수 있습니다. 따라서 동일한 전자상거래 데이터가 웹서버 로그와 ELF2 로그 둘 다에 존재할 경우 Urchin에서 중복 거래를 생성하게 되므로, 웹서버 로그에 동일한 데이터가 존재하면 ELF2 로그는 처리되지 않도록 하는 것이 중요합니다.

유틸리티

  • Urchin에서 제공되는 zip/unzip 유틸리티는 크기가 2GB 이하인 파일만 지원합니다. 이로 인해 Urchin에서 대규모 사이트의 보고 데이터에 대한 백업 및 아카이브를 생성하거나 로그 소스의 로그 저장 위치에 "압축" 옵션을 사용할 때 제한이 있을 수 있습니다.
  • udb-sanitizer에서 대용량 데이터가 포함된 프로필을 처리할 때 "즉시 실행" 기능의 처리 속도가 크게 저하될 수 있습니다.

보고

  • 대규모 트래픽이 발생하거나 기간이 긴 사이트에 대해 세그먼트별로 비교하는 보고서를 작성할 때 처리 속도가 크게 저하되거나 시간제한에 걸려 보고서가 생성되지 않을 수 있습니다.

외부 인증

  • Urchin 6.6+는 LDAP을 비롯한 외부 인증 모듈과 함께 제공됩니다. 현재 LDAP 모듈은 "SSL/TLS"를 운영하는 LDAP 제공업체와 연동되지 않습니다.

설치

  • 데모 라이센스를 사용 중인 Urchin 인스턴스를 업그레이드하는 것은 권장되지 않으며 로그인/라이센싱 문제가 발생할 수 있습니다.

쿠키

  • 보고 인터페이스의 기능을 향상시키기 위해 Urchin 6.6+에서는 클라이언트측 쿠키를 사용합니다. 따라서 클라이언트 브라우저에 클라이언트측 쿠키가 활성화되어 있어야 보고서를 제대로 볼 수 있습니다.

Urchin 6.600 - 개정: 2009년 5월 29일
