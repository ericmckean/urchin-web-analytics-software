Urchin 6.6、6.601 和 6.602 的已知問題

Urchin 6.602 - 修訂日期： 2009 年 10 月 23 日

管理

  • 「目標組態」精靈使用網站內點閱分佈圖技術，但該技術無法處理受密碼保護的網頁，而且在處理複雜的網頁 (例如包含頁框、javascript、css 或自動重新導向等技術的網頁) 時容易發生問題。

地理資料

  • 採用新的地理資料資料庫格式後，使用者無法再匯入本地 IP 至網域名稱對映資訊。

記錄處理作業

  • 若未按照記錄時間順序進行處理，高流量網站的記錄處理作業可能需時甚久。
  • 如果資料庫的記錄上限超出預設值 (100,000 筆)，可能會拖累記錄處理成效，對高流量網站的影響尤其明顯。
  • 對於處理過程中開啟的所有實體記錄檔，記錄處理引擎會分別替設定檔中的各個篩選器分配記憶體。 如果設定檔內含大量篩選器，而記錄檔來源又與大量的實體記錄檔比對成功 (例如使用過於籠統的廣泛萬用字元比對)，則會產生大量的記憶體配置要求，導致無法順利分配所有必要的記憶體。
  • 如果查閱表格規模太大，可能會導致記憶體配置需求量激增，導致記憶體配置失敗；在記錄處理過程中大量開啟實體記錄檔時，最常發生這種問題。

  • 使用 UNC 通訊端時，可能會發生某些已知問題。 詳情請參閱 UNC記錄來源使用說明。

排程器

  • Urchin 排程器 (設定檔和單次點擊出價) 使用資料庫來儲存並擷取用於安排各種工作的時間戳記。 因此，一旦資料庫配置完畢，就不能更改時區 (也就是說，不能更改執行 urchin 資料庫之電腦所在的時區)。 此外，極力建議您不要更改執行 Urchin 排程器之電腦所在的時區。

E-Commerce

  • 無論是透過 UTM javascript 呼叫記錄交易時直接內嵌在網路伺服器的 e-commerce 資料，還是由購物車建立的明確 ELF2 e-commerce 記錄，Urchin 都可以讀取。不過，如果網路伺服器記錄和 ELF2 記錄顯示了相同的 e-commerce 資料，會導致 Urchin 建立重複的交易，因此，如果網路伺服器記錄中已內嵌相同的資料，請千萬不要進行 ELF2 記錄處理作業。

Utilities

  • 如果設定檔內含大量資料，udb-sanitizer 的"「日期歸零」"功能執行速度可能十分緩慢。

報表

  • 如果網站流量太多和/或日期範圍太大，交叉分析報表的執行速度可能十分緩慢 (甚至發生作業逾時)。
  • "「媒介轉換」"報表與"「分析選項」"區段下所有子報表回報的"「網頁檢視/造訪次數」"結果不同。

與 AdWords 整合

  • 如果將 Urchin 6.6+ 安裝在 Windows 2008 64-bit 版上，就無法從 Urchin 直接存取 AdWords。

PostgreSQL 安裝

  • 在 Linux Kernel 2.6 和 Windows 上安裝 PostgreSQL 伺服器時，安裝程式需要 root/admin 權限才能建立名為 "postgres" 的系統使用者。
  • 在 Linux 上，PostgreSQL 安裝程式需要 glibc 2.3.3 或更新版本。
  • 在 Linux Kernel 2.4 上安裝 Urchin 時，如果選擇了隨附的 PostgreSQL 伺服器，可能會導致單次點擊出價資料匯入作業無法運作。 在此情況下，我們建議您使用外部 PostgreSQL 伺服器。

Urchin 6.601 - 修訂日期： 2009 年 7 月 23 日

管理

  • 在「資料中心」模式中，帳戶管理員只有一種權限層級，其限制比 Urchin 5 中的聯盟管理員權限更為嚴格。
  • 「目標組態」精靈使用網站內點閱分佈圖技術，但該技術無法處理受密碼保護的網頁，而且在處理複雜的網頁 (例如包含頁框、javascript、css 或自動重新導向等技術的網頁) 時容易發生問題。

地理資料

  • 採用新的地理資料資料庫格式後，使用者無法再匯入本地 IP 至網域名稱對映資訊。

記錄處理作業

  • 基於成效方面的考量，記錄處理引擎會在處理過程中將整個地理資料資料庫載入記憶體，如此一來，即使是處理規模很小的網站設定檔，也會佔用相當多的記憶體 (至少 800 MB)， 而且會造成每個設定檔的記錄處理初始化時間出現一定的延遲，通常為 2-5 秒。
  • 若未按照記錄時間順序進行處理，高流量網站的記錄處理作業可能需時甚久。
  • 如果資料庫的記錄上限超出預設值 (100,000 筆)，可能會拖累記錄處理成效，對高流量網站的影響尤其明顯。
  • 對於處理過程中開啟的所有實體記錄檔，記錄處理引擎會分別替設定檔中的各個篩選器分配記憶體。 如果設定檔內含大量篩選器，而記錄檔來源又與大量的實體記錄檔比對成功 (例如使用過於籠統的廣泛萬用字元比對)，則會產生大量的記憶體配置要求，導致無法順利分配所有必要的記憶體。
  • 如果查閱表格規模太大，可能會導致記憶體配置需求量激增，導致記憶體配置失敗；在記錄處理過程中大量開啟實體記錄檔時，最常發生這種問題。

排程器

  • Urchin 排程器 (設定檔和單次點擊出價) 使用資料庫來儲存並擷取用於安排各種工作的時間戳記。 因此，一旦資料庫配置完畢，就不能更改時區 (也就是說，不能更改執行 urchin 資料庫之電腦所在的時區)。 此外，極力建議您不要更改執行 urchin 排程器之電腦所在的時區。

E-Commerce

  • 無論是透過 UTM javascript 呼叫記錄交易時直接內嵌在網路伺服器的 e-commerce 資料，還是由購物車建立的明確 ELF2 e-commerce 記錄，Urchin 都可以讀取。不過，如果網路伺服器記錄和 ELF2 記錄顯示了相同的 e-commerce 資料，會導致 Urchin 建立重複的交易，因此，如果網路伺服器記錄中已內嵌相同的資料，請千萬不要進行 ELF2 記錄處理作業。

公用程式

  • Urchin 隨附的壓縮/解壓縮公用程式有 2 GB 的檔案大小上限。 這可能會讓 Urchin 無法為大型網站建立報表資料備份/封存，或是在處理記錄來源時無法使用「記錄命運」中的"「壓縮」"選項。
  • 如果設定檔內含大量資料，udb-sanitizer 的"「日期歸零」"功能執行速度可能十分緩慢。

報表

  • 如果網站流量太多和/或日期範圍太大，交叉分析報表的執行速度可能十分緩慢 (甚至發生作業逾時)。

外部驗證

  • Urchin 6.6+ 內建外部驗證模組，其中包含 LDAP。 LDAP 模組目前無法與透過 "SSL/TLS" 執行的 LDAP 供應商一起運作。

安裝

  • 使用 Urchin 試用授權時，最好不要進行升級作業，否則可能會導致登入/授權問題。

Cookie

  • 為了強化報表介面的互動方式，Urchin 6.6+ 使用用戶端 Cookie。 您必須在瀏覽器中啟用用戶端 Cookie，才能正確瀏覽報表。
