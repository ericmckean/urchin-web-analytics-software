Urchin 6.6, 6.601 et 6.602 : problèmes connus

Urchin 6.602 - Date de révision : 23 octobre 2009

Administration

  • L'assistant de configuration d'objectifs fait appel à la technologie de synthèse données/site, qui ne gère pas les pages protégées par mot de passe et montre rapidement ses limites avec des pages complexes (par exemple, les pages avec des cadres, des redirections automatiques, des éléments JavaScript, css, etc.).

Données géographiques

  • En raison du format de la nouvelle base de données géographique, il n'est plus possible d'importer des informations de mappage IP local-nom de domaine.

Traitement des journaux

  • Le traitement des journaux sans respecter l'ordre chronologique sera probablement très lent pour les sites enregistrant des niveaux élevés de trafic.
  • L'augmentation de la limite de la base de données au-delà des 100 000 enregistrements par défaut peut diminuer les performances de traitement des journaux, tout particulièrement avec les sites enregistrant des niveaux de trafic élevés.
  • Le moteur de traitement des journaux alloue de la mémoire à chaque filtre appliqué à un profil pour chaque occurrence de fichier journal ouvert lors du traitement. Ceci explique parfois les requêtes d'allocation de mémoire volumineuse et, dans certains cas, l'impossibilité d'allouer la mémoire nécessaire, lorsqu'un profil est configuré avec de nombreux filtres et qu'un nombre élevé de fichiers journaux correspondent aux sources de journal (en raison, par exemple, d'une recherche trop large avec caractère générique).
  • Des tableaux de conversion volumineux peuvent entraîner une allocation de mémoire élevée et, dans certains cas, des erreurs d'allocation. Ce cas de figure se produit notamment lorsqu'un grand nombre de fichiers journaux sont ouverts lors du traitement des journaux.

  • Il existe certains problèmes connus liés à l'utilisation des sockets UNC. Pour plus d'informations, consultez l'article Gestion des sources du journal UNC.

Planificateurs

  • Les planificateurs Urchin (profil et CPC) utilisent la base de données pour stocker et récupérer les valeurs de date et d'heure afin de planifier plusieurs tâches. Par conséquent, vous ne devez pas modifier les fuseaux horaires (c'est-à-dire modifier le fuseau horaire d'une machine exécutant la base de données Urchin) une fois la base de données configurée. Vous ne devez pas non plus modifier le fuseau horaire d'une machine exécutant les planificateurs Urchin.

Commerce électronique

  • Urchin peut parcourir les données de commerce électronique qui sont directement intégrées dans les journaux de serveur Web (lorsque les transactions sont consignées via des appels UTM javascript) ou via un journal explicite de commerce électronique ELF2 créé par le panier électronique. Urchin crée des transactions dupliquées si les mêmes données de commerce électronique sont présentes à la fois dans les journaux de serveur Web et dans les journaux ELF2. Par conséquent, il est important de ne pas traiter les journaux ELF2 si les mêmes données figurent dans les journaux de serveur Web.

Utilitaires

  • La fonctionnalité "jour zéro" dans udb-sanitizer peut être particulièrement lente pour les profils associés à des volumes de données élevés.

Création de rapports

  • La création de rapports par segmentation croisée peut être particulièrement lente (une expiration du délai d'exécution peut même se produire) pour les sites enregistrant des niveaux de trafic élevés et/ou avec des plages de dates étendues.
  • Les résultats "P/Visites" ne sont pas les mêmes si l'on compare le rapport "Conversion par support" à chaque sous-rapport situé sous "Options d'analyse".

Intégration avec AdWords

  • Il est impossible d'accéder directement à AdWords à partir d'Urchin si Urchin 6.6+ est installé sur Windows 2008 version 64-bit.

Installation de PostgreSQL

  • Lors de l'installation du serveur PostgreSQL sur Linux Kernel 2.6 et Windows, le programme d'installation nécessite des privilèges d'administrateur root/admin pour créer l'utilisateur système "postgres".
  • Sur Linux, le programme d'installation PostgreSQL requiert glibc 2.3.3 ou une version ultérieure.
  • Il est possible que l'importation des données CPC ne fonctionne pas si le serveur PostgreSQL fourni est sélectionné lors de l'installation d'Urchin sur Linux Kernel 2.4. Dans ce cas, nous vous recommandons d'utiliser un serveur PostgreSQL externe.

Urchin 6.601 - Date de révision : 23 juillet 2009

Administration

  • En mode Data Center, les administrateurs de compte ne disposent que d'un seul type de droit, qui est plus restrictif que les droits des administrateurs affiliés disponibles dans Urchin 5.

  • L'assistant de configuration d'objectifs fait appel à la technologie de synthèse données/site, qui ne gère pas les pages protégées par mot de passe et montre rapidement ses limites avec des pages complexes (par exemple, les pages avec des cadres, des redirections automatiques, des éléments JavaScript, css, etc.).

Données géographiques

  • En raison du format de la nouvelle base de données géographique, il n'est plus possible d'importer des informations de mappage IP local-nom de domaine.

Traitement des journaux

  • Pour garantir des résultats optimaux, le moteur de traitement des journaux charge l'intégralité de la base de données géographique en mémoire. Ceci explique l'empreinte mémoire relativement volumineuse (environ 800 Mo au minimum) et ce, même lorsque les profils traités appartiennent à des sites de taille réduite. Il faut également signaler un court temps de latence en début de traitement des journaux pour chaque profil (habituellement compris entre 2 et 5 secondes).
  • Le traitement des journaux sans respecter l'ordre chronologique sera probablement très lent pour les sites enregistrant des niveaux élevés de trafic.
  • L'augmentation de la limite de la base de données au-delà des 100 000 enregistrements par défaut peut diminuer les performances de traitement des journaux, tout particulièrement avec les sites enregistrant des niveaux de trafic élevés.
  • Le moteur de traitement des journaux alloue de la mémoire à chaque filtre appliqué à un profil pour chaque occurrence de fichier journal ouvert lors du traitement. Ceci explique parfois les requêtes d'allocation de mémoire volumineuse et, dans certains cas, l'impossibilité d'allouer la mémoire nécessaire, lorsqu'un profil est configuré avec de nombreux filtres et qu'un nombre élevé de fichiers journaux correspondent aux sources de journal (en raison, par exemple, d'une recherche trop large avec caractère générique).
  • Des tableaux de conversion volumineux peuvent entraîner une allocation de mémoire élevée et, dans certains cas, des erreurs d'allocation. Ce cas de figure se produit notamment lorsqu'un grand nombre de fichiers journaux sont ouverts lors du traitement des journaux.

Planificateurs

  • Les planificateurs Urchin (profils et CPC) utilisent la base de données pour stocker et récupérer les valeurs de date et d'heure afin de planifier plusieurs tâches. Par conséquent, vous ne devez pas modifier les fuseaux horaires (c'est-à-dire modifier le fuseau horaire d'une machine exécutant la base de données Urchin) une fois la base de données configurée. Vous ne devez pas non plus modifier le fuseau horaire d'une machine exécutant les planificateurs urchin.

Commerce électronique

  • Urchin peut lire les données de commerce électronique qui sont directement intégrées dans les journaux du serveur Web (lorsque les transactions sont consignées via des appels UTM JavaScript) ou via un journal de commerce électronique ELF2 explicite créé par le panier électronique. Urchin crée des transactions dupliquées si les mêmes données de commerce électronique sont présentes à la fois dans les journaux du serveur Web et dans les journaux ELF2. Par conséquent, il est important de ne pas traiter les journaux ELF2 si les mêmes données figurent dans les journaux du serveur Web.

Utilitaires

  • La taille maximale autorisée pour les fichiers traités par les utilitaires de compression et de décompression fournis avec Urchin est de 2 Go. Cette limite peut avoir une incidence sur la capacité d'Urchin à créer des sauvegardes/archives des données de rapport pour les sites volumineux ou à utiliser la fonction de "compression" avec l'option "Avenir du journal" pour les sources de journal.
  • La fonctionnalité "jour zéro" de udb-sanitizer peut être particulièrement lente pour les profils associés à des volumes de données élevés.

Création de rapports

  • La création de rapports par segmentation croisée peut être particulièrement lente (une expiration du délai d'exécution peut même se produire) pour les sites enregistrant des niveaux de trafic élevés et/ou avec des plages de dates étendues.

Authentification externe

  • Urchin 6.6+ est fourni avec des modules d'authentification externe, notamment LDAP. Dans l'immédiat, le module LDAP ne fonctionne pas avec les fournisseurs LDAP qui s'exécutent sur "SSL/TLS".

Installation

  • Il est déconseillé de mettre à niveau une instance d'Urchin qui utilise une licence de démo, car cela peut entraîner des problèmes de connexion/licence.

Cookies

  • Pour améliorer l'interaction dans l'interface de création de rapports, Urchin 6.6+ utilise des cookies côté client. Pour un affichage optimal des rapports, les cookies côté client doivent être activés dans le navigateur du client.

  • Si vous tentez de vous connecter à un compte qui comprend une quantité importante de données, l'erreur suivante peut s'afficher : "Erreur de session – Appel du programme par défaut impossible. (2002) Veuillez vous reconnecter". Ce problème se produit lorsque l'extraction des données pour des rapports d'ensemble dépasse le délai d'expiration par défaut fixé à 45 secondes (APPLICATION_TIMEOUT). Pour contourner ce problème, augmentez la valeur APPLICATION_TIMEOUT dans le fichier session.conf en la définissant sur 180 (ou plus).
