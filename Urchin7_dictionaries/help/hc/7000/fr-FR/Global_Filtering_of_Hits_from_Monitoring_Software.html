Filtrage global des appels de fichier provenant du logiciel de contrôle

Présentation

La plupart des environnements hôte assurent la surveillance des serveurs Web clients de manière à respecter leurs accords de niveau de service. Toutefois, les appels de fichier générés par cette procédure de contrôle peuvent avoir une incidence importante sur les rapports Urchin relatifs aux sites Web surveillés, provoquant une augmentation artificielle du nombre de sessions, de consultations de page, d'appels de fichiers et d'octets

Recommandation

Dans les environnements d'hébergement qui appliquent ce type de surveillance, il est vivement recommandé d'appliquer un filtre Urchin standard ou global aux profils Urchin configurés par chaque client, afin d'écarter tout appel de fichier généré par le logiciel de contrôle. Cela peut être facilement mis en œuvre dans un environnement où l'installation Urchin est centralisée (gérée par une société d'hébergement) et fournit des rapports distincts pour chaque site Web de client. Dans les environnements dédiés ou de colocalisation, dans lesquels les clients gèrent leur instance d'Urchin sur leur(s) serveur(s), la société d'hébergement doit fournir un modèle adapté pour réaliser le filtrage des contrôles effectués.

Pour faciliter la mise en place du filtrage Urchin, l'hôte et le client peuvent décider de créer une page spécifique sur le site Web du client, destinée exclusivement au logiciel de contrôle, par exemple : 

     http://www.customerdomain.com/healthpage.html

Exemples

Exemple 1 : Exclure l'adresse IP du système de contrôle 

      Filter Type: Exclude
      Filter Field: IP
      Filter Spec: 172\.16\.1\.1
  • Cette entrée permet à Urchin d'éliminer tous les appels de fichier provenant de l'adresse IP 172.16.1.1 lors du traitement du journal de serveur Web.

Exemple 2 : Exclure la page appelée par le système de contrôle 

      Filter Type: Exclude
      Filter Field: REQUEST
      Filter Spec: ^/healthpage.html
  • Cette entrée permet à Urchin d'éliminer toutes les demandes visant la page /healthpage.html lors du traitement du journal de serveur Web.

Considérations

  1. Il peut être souhaitable de créer d'autres profils non filtrés, afin que les clients puissent connaître la charge réelle de trafic sur le ou les serveurs Web (y compris les données filtrées).
  2. La société d'hébergement peut fournir un profil traitant exclusivement les appels de fichier liés à la surveillance, incluant uniquement les appels de fichier provenant du logiciel de contrôle. Ce profil prouvera que des contrôles adaptés sont effectués, conformément aux accords de niveau de service passés.
