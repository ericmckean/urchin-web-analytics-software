Vue d'ensemble des fonctionnalités d'intégration Urchin

Introduction

Le frontal Urchin est constitué de composants modulaires et de fonctions standard permettant aux administrateurs d'intégrer l'accès aux rapports Urchin dans les infrastructures existantes et à venir. Afin de respecter notre engagement auprès des centres de données et des fournisseurs d'hébergement, nous testons et prenons en charge trois points qui permettent d'intégrer Urchin dans la majorité des architectures.

Le schéma suivant présente les principaux composants du frontal Urchin ainsi que ses trois points d'intégration. L'administration et la création de rapports sont assurées par des fonctions Web fournies à partir de ce système.

Les contenus provenant du système Urchin sont distribués via un serveur Apache installé sous le nom "urchinwebd", comme indiqué à gauche sur le schéma. Les demandes sont gérées par le serveur Apache et transmises à l'aide de l'interface CGI à un logiciel contrôleur de sessions (session.cgi). Le contrôleur de session réalise l'authentification et transmet la demande soit au moteur d'administration (admin), soit au moteur de création de rapports Urchin (urchin.cgi), selon l'action demandée. Le contenu est renvoyé à l'utilisateur avec un identifiant de session intégré.

Les lettres "A", "B" et "C" désignent les points d'intégration mentionnés précédemment. Le point "A" permet à l'administrateur de remplacer ou de contourner le serveur Web Urchin. Le point "B" autorise les authentifications externes ou le contournement de l'authentification. Enfin, le point "C" donne la possibilité de fournir un accès direct aux rapports via une classe enveloppante (wrapper) ou un portail.

Point A : Intégration au serveur Web

La plupart des sociétés d'hébergement disposent d'un serveur Web opérationnel ou d'un serveur Web spécialement compilé pour leur système. Si ce serveur Web assure les principales fonctions CGI, vous pouvez remplacer le binaire Apache ou utiliser un serveur Web existant pour fournir un accès à Urchin.

Ce point d'intégration n'a aucune incidence sur le reste du système Urchin. L'interface Urchin sera utilisée pour les opérations d'administration, d'authentification et de distribution des rapports. Vous devrez tout de même configurer les utilisateurs et profils Urchin de sorte que le logiciel puisse identifier les droits d'accès des utilisateurs entrant dans le système afin de leur présenter les rapports adéquats.

Point B : Authentification externe ou contournement de l'authentification

Les environnements d'hébergement complexes disposent généralement de protocoles centralisés assurant l'authentification des utilisateurs, tels que LDAP. Une simple modification de votre configuration permettra au contrôleur de session d'utiliser la tâche d'authentification de votre choix. Une interface simple permet de renvoyer les connexions réussies et échouées. Les tâches d'authentification existantes sont réunies dans un seul programme offrant une structure adaptée aux deux systèmes. En utilisant l'intégration au niveau du point "B", il est également possible de contourner l'authentification à l'aide d'une tâche factice et d'établir un lien direct vers la page de destination.

L'intégration au point "B" préserve l'intégrité du système Urchin dans sa globalité, à l'exception de l'authentification qui est exécutée par une application externe. Vous devrez tout de même configurer les utilisateurs et profils Urchin de sorte que le logiciel puisse identifier les droits d'accès des utilisateurs entrant dans le système afin de leur présenter les rapports adéquats. Pour consulter une description détaillée de l'interface d'authentification externe et un exemple de contournement de l'authentification, veuillez vous reporter à la rubrique Authentification externe ou contournement de l'authentification.

Point C : Liens directs aux rapports via une classe enveloppante ou un portail

Nombreux sont nos clients qui instituent une relation individualisée entre le client et le site Web, et qui souhaitent établir un lien direct aux rapports d'un site donné à partir de sa zone d'administration. En général, les utilisateurs accédant à cette zone sont déjà authentifiés. L'intégration au niveau du point C facilite la création d'un lien direct vers les rapports à l'aide d'un script enveloppant ou d'un portail existant. Dans cette configuration, l'authentification Urchin et l'écran initial de sélection des rapports sont contournés et les utilisateurs sont orientés directement vers les rapports Urchin.

L'intégration au niveau du point "C" implique que le fournisseur de service contrôle les droits d'accès des utilisateurs, par le biais d'une relation individualisée ou d'une base de données de configuration existante. La configuration d'Urchin reste nécessaire pour la génération des rapports de profil, mais il est possible d'automatiser cette opération dans l'interface d'administration ou en dehors de celle-ci. Urchin offre la possibilité de créer un lien direct vers un rapport Urchin à l'aide d'une URL spécifique lorsque l'option "Liens directs aux rapports" est activée dans l'interface d'administration. L'accès au rapport Urchin est contrôlé par le fichier ".report.conf" situé dans le répertoire spécifié par cette URL.

Dans le cadre d'une intégration à des portails complexes, le moteur de création de rapports Urchin transmet la session et d'autres variables du portail afin de maintenir une session opérationnelle.
