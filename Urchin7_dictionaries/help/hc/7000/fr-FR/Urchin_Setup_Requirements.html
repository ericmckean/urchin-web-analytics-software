Contraintes de configuration pour Urchin 6

  • Cet article a été rédigé par EpikOne, un consultant agréé du logiciel Urchin.

Cet article répertorie les problèmes de fonctionnement à anticiper avant l'installation et l'exécution d'Urchin. Certaines de ces informations sont indispensables au bon fonctionnement d'Urchin. D'autres éléments sont importants dans l'optique d'une utilisation optimale d'Urchin une fois le logiciel installé.

Installation d'Urchin - Considérations de base

  • Sous Windows, vous devez vous connecter en tant qu'administrateur pour procéder à l'installation.
  • Sur les systèmes de type UNIX, vous pouvez effectuer l'installation en tant que simple utilisateur. Sachez toutefois que si vous n'effectuez pas l'installation en tant que superutilisateur, vous ne serez pas autorisé à procéder à l'installation dans tous les secteurs du système de fichiers.
  • Urchin est fourni en kit avec un serveur Web Apache (binaire) pour la configuration et la diffusion des rapports. Vos administrateurs système doivent savoir que ce nouveau service Web sera en cours d'exécution après l'installation d'Urchin.
  • Vous devez installer Urchin 6 sur un disque disposant d'un espace libre important (plusieurs centaines de méga-octets au minimum) en prévision de la croissance du volume des bases de données Urchin. Pour en savoir plus, consultez la section Problèmes de performance et de gestion.

  • Si vous effectuez une mise à niveau à partir d'Urchin 5, vous devez importer vos bases de données dans Urchin 6 à l'aide des utilitaires uconf2sql et convert-u5data. Vous ne pouvez pas effectuer une mise à niveau directe d'Urchin 5 à Urchin 6 en exécutant simplement le programme d'installation Urchin 6. Consultez la rubrique relative aux mises à niveau de la section Mise en route du Centre de documentation.

  • La mise à niveau d'Urchin 5 à Urchin 6 exige l'obtention d'une nouvelle licence.

Traitements Urchin - Considérations de base

  • Accès aux journaux des serveurs Web : vous devez connaître le chemin d'accès aux fichiers journaux d'un site donné et vous devez être autorisé à accéder à ces fichiers. Si les journaux se trouvent sur un système distant, vous devez disposer d'un compte et d'un mot de passe pour les récupérer (FTP/HTTP/HTTPS/UNC).
  • Format de journal adéquat : bien qu'Urchin puisse traiter des formats de journaux personnalisés, vous pouvez simplifier les opérations de gestion si vous configurez votre serveur Web de sorte que les journaux apparaissent dans un format standard. Nous vous recommandons d'utiliser le format de journal combiné étendu (par exemple, les journaux NCSA ou Apache) ou le format étendu W3C (par exemple, les journaux IIS). Pour les sites IIS, vous devez désactiver la consignation de la comptabilité des traitements. Pour des informations supplémentaires, consultez la section Configuration avancée.
  • Compte d'utilisateur unique pour les traitements Urchin : sur les systèmes de type UNIX, il est préférable de renforcer la sécurité en exécutant les programmes Urchin sous un ID utilisateur spécial utilisé exclusivement pour Urchin et disposant de droits restreints. La configuration d'un tel compte exige que vous disposiez de droits particuliers ou de droits de superutilisateur.
  • Planification : pour générer des rapports en temps voulu et intégrer le temps nécessaire au traitement d'ensembles de données de taille importante, vous devez planifier l'exécution des traitements Urchin.

Traitements Urchin - Considérations avancées

  • Si vous souhaitez bénéficier du suivi des visiteurs uniques, vous devez appliquer la procédure de base suivante :
    • Installez le code de détection UTM sur les pages Web de votre site.
    • Activez la consignation des cookies dans le format de journal pour votre serveur Web.
    • Dans le profil Urchin, définissez la méthode de suivi UTM pour le site Web.
  • Si vous décidez de ne pas utiliser le suivi des visiteurs uniques, vous devez choisir le niveau de détail de vos rapports sur les visiteurs ou les sessions. Vous devez également sélectionner une méthode de suivi des visiteurs appropriée pour chaque site. La seule autre méthode de suivi disponible dans Urchin 6 est IP + User-Agent.

Problèmes de performance et de gestion

  • Rotation des journaux : si vous ne disposez pas d'un mécanisme externe pour archiver ou supprimer les journaux de serveur Web une fois qu'ils ont été traités par Urchin, vous pouvez configurer Urchin (via les paramètres avancés de chaque source de journal) pour qu'il effectue cette tâche.
  • Conservation des bases de données Urchin pour les rapports sur l'historique : une fois les bases de données d'un mois spécifique créées, elles sont disponibles, à l'avenir, pour d'éventuelles analyses d'historique. Les utilisateurs doivent décider jusqu'où remontent leurs données d'historique à conserver afin de planifier la purge des données inutiles pour économiser de l'espace disque. Vous pouvez configurer Urchin de sorte qu'il compresse les bases de données antérieures à une date spécifique.
  • Mémoire requise : Urchin est doté de systèmes de contrôle de la configuration permettant de limiter la quantité de RAM utilisée lors du traitement des journaux. Le paramètre par défaut est défini sur 128 Mo, ce qui peut s'avérer insuffisant pour les sites dont les journaux sont d'une taille supérieure à 10 Mo. Prévoyez de disposer d'une quantité de mémoire RAM suffisante pour être en mesure d'augmenter l'utilisation de mémoire d'Urchin en fonction de vos besoins et d'ajuster les paramètres de mémoire du logiciel pour optimiser les performances de traitement.
  • Emplacement du stockage des données Urchin : à l'aide du fichier etc/urchin.conf, vous pouvez configurer Urchin de sorte que les bases de données de rapports soient stockées dans un système de fichiers externe à la distribution Urchin. Cela vous permet d'attribuer, à l'emplacement le plus pratique, suffisamment d'espace dédié au système de fichiers en prévision de la croissance des bases de données.

Problèmes d'accès à distance et d'intégration

  • Utilisation de SSL pour les tâches d'administration et de rapports Urchin : le serveur Web Apache livré avec Urchin est compilé de sorte à prendre en charge SSL. SSL n'est pas activé dans la configuration par défaut. Toutefois, l'utilisateur peut l'activer s'il le souhaite.
  • Configuration du pare-feu : si votre topologie réseau inclut des pare-feu, des serveurs proxy et d'autres éléments placés entre le serveur de traitement Urchin et des utilisateurs qui tentent de consulter des rapports ou des systèmes comportant des journaux à extraire, ces périphériques devront être configurés de sorte qu'ils ne perturbent pas l'accès à distance d'Urchin. Vous pouvez normalement y parvenir sans compromettre la sécurité qu'une topologie de ce type est justement censée fournir.
