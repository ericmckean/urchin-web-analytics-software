Urchin 6.6, 6.601 和 6.602 已知问题

Urchin 6.602 - 修订日期： 2009 年 10 月 23 日

管理

  • “目标配置”向导使用的网站内点击量分布图技术无法处理受密码保护的网页，而且容易受复杂网页（例如包含框架的网页、javascript 网页、css 网页或自动重定向网页）干扰。

地理数据

  • 受新地理数据库的格式影响，您现在无法导入本地“IP 至域”名称映射信息。

日志处理

  • 如果不按时间顺序处理日志，则可能会导致高流量网站的日志处理过程非常缓慢。
  • 如果数据库的容量限制超过了默认的 100000 条记录，则可能会使日志处理性能受到影响，对于高流量的网站尤其如此。
  • 对于处理过程中打开的每个物理日志文件，日志处理引擎会对应用于配置文件的每个过滤器分配内存。 如果配置文件具有大量过滤器，并且相应日志源产生了大量的匹配物理日志文件（例如由于过于笼统的通配符匹配），这可能会产生大量内存分配请求，进而导致无法顺利分配所有必要的内存。
  • 如果查找表规模过大，则会导致内存分配要求激增，并可能会造成分配失败；特别是在日志处理过程中打开大量物理日志文件时更是如此。

  • 使用 UNC 套接字时可能会遇到一些问题。 要了解详情，请参阅使用 UNC 日志源。

调度程序

  • Urchin 调度程序（配置文件和每次点击费用）会使用数据库来存储和检索用于安排各种工作的时间戳。 因此，数据库在配置好后时区设置不应再发生更改（即不要更改运行 urchin 数据库的计算机的时区）。 此外，强烈建议您不要更改运行 Urchin 调度程序的计算机的时区。

电子商务

  • 无论对于通过 UTM javascript 调用在记录交易时直接嵌入在网络服务器日志中的电子商务数据，还是由购物车创建的显式 ELF2 电子商务日志，Urchin 都可以读取。如果网络服务器日志和 ELF2 日志显示的是相同的电子商务数据，则 Urchin 会创建重复的交易，因此，如果已在网络服务器日志中嵌入了相同的数据，则请务必不要对 ELF2 日志进行处理。

实用程序

  • 对于包含了大量数据的配置文件，使用 udb-sanitizer 的“零天”(zero day) 功能可能会导致执行速度非常缓慢。

报告

  • 如果网站流量过大和/或日期范围过长，跨群体细分报告功能的执行速度可能十分缓慢（甚至发生操作超时）。
  • “媒介转化”报告与“分析选项”下各个子报告的“每次访问浏览量”结果不同。

与 AdWords 的整合

  • 如果将 Urchin 6.6+ 安装在 Windows 2008 的 64-bit 版本上，就无法从 Urchin 直接访问 AdWords。

PostgreSQL 安装

  • 在 Linux Kernel 2.6 和 Windows 上安装 PostgreSQL 服务器时，安装程序需要具有 root/admin 权限才能创建名为 "postgres" 的系统用户。
  • 在 Linux 上，PostgreSQL 安装程序需要 glibc 2.3.3 或更高版本。
  • 在 Linux Kernel 2.4 上安装 Urchin 时，如果选择了附带的 PostgreSQL 服务器，可能会导致每次点击费用数据无法正常导入。 在这种情况下，建议您使用外部 PostgreSQL 服务器。

Urchin 6.601 - 修订日期： 2009 年 7 月 23 日

管理

  • 在“数据中心”模式下，帐户管理员只有一种权限级别，而且所受限制比 Urchin 5 中的关联企业管理员权限级别更为严格。
  • “目标配置”向导使用的网站内点击量分布图技术无法处理受密码保护的网页，而且容易受复杂网页（例如包含框架的网页、javascript 网页、css 网页或自动重定向网页）干扰。

地理数据

  • 受新地理数据库的格式影响，您现在无法导入本地“IP 至域”名称映射信息。

日志处理

  • 出于性能方面的考虑，日志处理引擎会在处理过程中将整个地理数据库载入内存，而这会导致即便是处理规模很小的网站的配置文件，内存占用量也会相当高（至少约为 800MB）。 另外，这也会使每个配置文件的日志处理初始化时间出现一定延迟，通常为 2 到 5 秒。
  • 如果不按时间顺序处理日志，则可能会导致高流量网站的日志处理过程非常缓慢。
  • 如果数据库的容量限制超过了默认的 100000 条记录，则可能会使日志处理性能受到影响，对于高流量的网站尤其如此。
  • 对于处理过程中打开的每个物理日志文件，日志处理引擎会对应用于配置文件的每个过滤器分配内存。 如果配置文件具有大量过滤器，并且相应日志源产生了大量的匹配物理日志文件（例如由于过于笼统的通配符匹配），这可能会产生大量内存分配请求，进而导致无法顺利分配所有必要的内存。
  • 如果查找表规模过大，则会导致内存分配要求激增，并可能会造成分配失败；特别是在日志处理过程中打开大量物理日志文件时更是如此。

调度程序

  • Urchin 调度程序（配置文件和每次点击费用）会使用数据库来存储和检索用于安排各种工作的时间戳。 因此，数据库在配置好后时区设置不应再发生更改（即不要更改运行 urchin 数据库的计算机的时区）。 此外，强烈建议您不要更改运行 urchin 调度程序的计算机的时区。

电子商务

  • 无论对于通过 UTM javascript 调用在记录交易时直接嵌入在网络服务器日志中的电子商务数据，还是由购物车创建的显式 ELF2 电子商务日志，Urchin 都可以读取。如果网络服务器日志和 ELF2 日志显示的是相同的电子商务数据，则 Urchin 会创建重复的交易，因此，如果已在网络服务器日志中嵌入了相同的数据，则请务必不要对 ELF2 日志进行处理。

实用程序

  • Urchin 提供的压缩/解压缩实用程序要求文件大小不得超过 2GB。 这可能会影响 Urchin 为大型网站创建报告数据备份/存档，或在针对日志源处置日志时使用“压缩”选项。
  • 对于包含了大量数据的配置文件，使用 udb-sanitizer 的“零天”(zero day) 功能可能会导致执行速度非常缓慢

报告

  • 如果网站流量过大和/或日期范围过长，跨群体细分报告功能的执行速度可能十分缓慢（甚至发生操作超时）。

外部身份验证

  • Urchin 6.6+ 包含外部身份验证模块，其中包括 LDAP。 目前，LDAP 模块不能用于运行在 "SSL/TLS" 之上的 LDAP 提供程序。

安装

  • 我们不推荐使用演示许可升级 Urchin 实例，这样做可能会导致登录/许可问题。

Cookie

  • 为便于在报告界面中进行更好的互动，Urchin 6.6+ 采用了客户端 Cookie。 在客户端浏览器中启用客户端 Cookie 后才能正确查看报告。
