監視ソフトウェアによるヒットのグローバル フィルタリング

概要

多くのホスティング環境では、顧客のウェブサーバーを監視することでサービス レベル 契約 (SLA) を維持しています。 しかし、このような監視によるヒットは、セッション数、ページビュー数、ヒット数、バイト数を人為的に増加させ、監視したウェブサイトの Urchin レポートが歪曲されてしまうという悪影響を伴います。

アドバイス

このような監視を行うホスティング環境の場合は、標準/グローバル Urchin フィルタを各顧客の Urchin 設定プロファイルに適用して、監視ソフトウェアによって生成されるヒット数を除外することをおすすめします。 ホスティング会社が一元管理する Urchin インストールにより各顧客のウェブサイトのレポートが提供される場合、この操作は簡単に行うことができます。 顧客自身が独自のサーバーで Urchin のインスタンスを管理する専用環境やコロケーション環境の場合は、ホスティング会社が行っている監視に適したサンプル フィルタを提供してもらう必要があります。

Urchin フィルタリングを導入するには、ホスティング会社と顧客が連携し、顧客のウェブサイト上に監視ソフトウェアのみが使用する専用ページを作成する必要があります。たとえば次のようになります。 

     http://www.customerdomain.com/healthpage.html

例

例 1: 監視システムの IP アドレスの除外 

      Filter Type: Exclude
      Filter Field: IP
      Filter Spec: 172\.16\.1\.1
  • この場合、IP アドレス 172.16.1.1 によるヒットは、Urchin が処理しているものとしてウェブサーバー ログから除外されます。

例 2: 監視システムがヒットする特定のページの除外 

      Filter Type: Exclude
      Filter Field: REQUEST
      Filter Spec: ^/healthpage.html
  • この場合、/healthpage.html をリクエストするヒットは、Urchin が処理しているものとしてウェブサーバー ログから除外されます。

アドバイス

  1. ウェブサーバーの除外ヒットを含む実際のトラフィックの負荷を確認できるよう、フィルタを適用しないプロファイルを顧客用に作成することをおすすめします。
  2. ホスティング会社は、監視ソフトウェアによるヒットのみをフィルタリングするなど、監視ヒット専用のレポートを提供するプロファイルを用意することをおすすめします。 このプロファイルは、監視が適切に行われ、サービス レベル契約の条件を満たしていることを示すために使用できます。
