Urchin 6.6、6.601、6.602 で報告されている問題

Urchin 6.602 - 更新日: 2009 年 10 月 23 日

管理

  • 目標設定ウィザードではサイト上のデータ表示技術を使用しているため、パスワードで保護されているページや複雑なページ（フレーム、javascript、css、自動リダイレクトなどを使用しているページ）を適切に処理できません。

地域データ

  • 新しいフォーマットの地域データ データベースを使用しているため、ローカルの IP とドメイン名のマッピング情報をインポートできません。

ログ処理

  • 時系列順でログを処理しない場合、トラフィックの多いサイトではログ処理に長い時間がかかる可能性があります。
  • データベースの上限をデフォルトの 100,000 以上に増やすと、特にトラフィックの多いサイトでパフォーマンスが低下する場合があります。
  • ログ処理エンジンは、処理中にログ ファイルが物理的に開かれるプロファイルに適用された各フィルタにメモリを割り当てます。 このため、プロファイルに多くのフィルタが設定されていて、ワイルドカードによる部分一致が多数含まれるなど、ログ ソースと一致する物理的なログファイルの数が多い場合は、大量のメモリ割り当てがリクエストされ、必要なすべてのメモリが割り当てられなくなる場合があります。
  • サイズの大きいルックアップ テーブルでは大量のメモリが割り当てられるため、ログの処理中に多数の物理的なログファイルを開く場合は特に、メモリの割り当てに失敗する場合があります。

  • 報告されている問題の中には、UNC ソケットの使用中に発生するものもあります。 詳細については UNC ログ ソースの設定 をご覧ください。

スケジューラ

  • Urchin では、データベースを使用してタイムスタンプを保存、取得し、スケジューラ、プロファイル、CPC の各種ジョブをスケジュール実行します。 このため、データベースの設定後は、タイムゾーン（urchin db を実行しているコンピュータのタイムゾーンなど）を変更しないでください。 また、Urchin スケジューラを実行しているコンピュータのタイムゾーンも変更しないでください。

e コマース

  • UTM javascript の呼び出しや、ショッピング カードで作成された専用の ELF2 e コマースのログによりトランザクションが記録される場合、Urchin ではウェブサーバーのログに直接埋め込まれた e コマース データを読み取ることができます。ウェブサーバーのログと ELF2 のログのどちらにも同じ e コマース データが存在する場合、Urchin で重複するトランザクションが作成されるため、ウェブサーバーのログに同じデータが埋め込まれている場合は ELF2 のログを処理しないようにすることが重要です。

ユーティリティ

  • udb-sanitizer の "日単位のデータ削除" 機能を使用すると、大量のデータが含まれるプロファイルの処理速度が遅くなる場合があります。

レポート

  • セグメント別のレポートでは、トラフィックが多いサイトや期間を長く設定した場合に、処理速度が遅くなったりタイムアウトする場合があります。
  • "1 回のセッションあたりの平均ページビュー数" は、"メディアのコンバージョン" レポートと、"Analysis オプション" の下にある各サブ レポートで異なります。

AdWords との統合

  • Urchin 6.6+ を 64 bit 版の Windows 2008 にインストールしている場合、Urchin から AdWords には直接アクセスできません。

PostgreSQL インストール

  • PostgreSQL サーバーを Linux Kernel 2.6 や Windows に インストールする場合、システム ユーザー "postgres" を作成するためにインストーラは root/admin 権限が必要です。
  • Linux で PostgreSQL をインストールするには glibc 2.3.3 以降が必要です。
  • Linux Kernel 2.4 での Urchin インストールでバンドルされた PostgreSQL サーバーを選択すると、CPC データをインポートできなくなる場合があります。 この場合は、外部の PostgreSQL サーバーを使用することをおすすめします。

管理

  • データセンター モードでは、アカウント管理者は 1 つの権限レベルしか持てないため、Urchin 5 のアフィリエーション管理者の権限レベルと比べて制限されます。
  • 目標設定ウィザードではサイト上のデータ表示技術を使用しているため、パスワードで保護されているページや複雑なページ（フレーム、javascript、css、自動リダイレクトなどを使用しているページ）を適切に処理できません。

地域データ

  • 新しいフォーマットの地域データ データベースを使用しているため、ローカルの IP とドメイン名のマッピング情報をインポートできません。

ログ処理

  • パフォーマンスを高めるため、ログ処理エンジンでは処理の際にメモリの地域データ データベース全体を読み取り、ごく小規模なサイトのプロファイルを処理する際もメモリ フットプリントが大きくなります（最低 800 MB）。 このため、各プロファイルのログ処理の初期化にかかる時間が若干（通常は 2～5 秒）長くなります。
  • 時系列順でログを処理しない場合、トラフィックの多いサイトではログ処理に長い時間がかかる可能性があります。
  • データベースの上限をデフォルトの 100,000 以上に増やすと、特にトラフィックの多いサイトでパフォーマンスが低下する場合があります。
  • ログ処理エンジンは、処理中にログ ファイルが物理的に開かれるプロファイルに適用された各フィルタにメモリを割り当てます。 このため、プロファイルに多くのフィルタが設定されていて、ワイルドカードによる部分一致が多数含まれるなど、ログ ソースと一致する物理的なログファイルの数が多い場合は、大量のメモリ割り当てがリクエストされ、必要なすべてのメモリが割り当てられなくなる場合があります。
  • サイズの大きいルックアップ テーブルでは大量のメモリが割り当てられるため、ログの処理中に多数の物理的なログファイルを開く場合は特に、メモリの割り当てに失敗する場合があります。

スケジューラ

  • Urchin では、データベースを使用してタイムスタンプを保存、取得し、スケジューラ、プロファイル、CPC の各種ジョブをスケジュール実行します。 このため、データベースの設定後は、タイムゾーン（urchin db を実行しているコンピュータのタイムゾーンなど）を変更しないでください。 また、urchin スケジューラを実行しているコンピュータのタイムゾーンも変更しないでください。

e コマース

  • UTM javascript の呼び出しや、ショッピング カードで作成された専用の ELF2 e コマースのログによりトランザクションが記録される場合、Urchin ではウェブサーバーのログに直接埋め込まれた e コマース データを読み取ることができます。ウェブサーバーのログと ELF2 のログのどちらにも同じ e コマース データが存在する場合、Urchin で重複するトランザクションが作成されるため、ウェブサーバーのログに同じデータが埋め込まれている場合は ELF2 のログを処理しないようにすることが重要です。

ユーティリティ

  • Urchin の zip/unzip ユーティリティでは、ファイル サイズが 2 GB に制限されます。 この制限は、Urchin で大規模なサイトのレポート データのバックアップやアーカイブを作成したり、ログ ソースで [ログの処置] の "圧縮" オプションを使用する場合に影響することがあります。
  • udb-sanitizer の "日単位のデータ削除" 機能を使用すると、大量のデータが含まれるプロファイルの処理速度が遅くなる場合があります。

レポート

  • セグメント別のレポートでは、トラフィックが多いサイトや期間を長く設定した場合に、処理速度が遅くなったりタイムアウトする場合があります。

外部認証

  • Urchin 6.6+ には、LDAP を含む外部認証モジュールが組み込まれています。 現在、この LDAP モジュールは "SSL/TLS" を使用する LDAP プロバイダでは機能しません。

インストール

  • デモ ライセンスを使用している Urchin はアップグレードしないでください。アップグレードするとログインやライセンスに関する問題が発生する可能性があります。

Cookie

  • レポート インターフェースの操作性を高めるため、Urchin 6.6+ ではクライアント サイドの Cookie を使用します。 レポートを適切に表示するには、クライアントのブラウザでクライアント サイドの Cookie が有効になっている必要があります。
