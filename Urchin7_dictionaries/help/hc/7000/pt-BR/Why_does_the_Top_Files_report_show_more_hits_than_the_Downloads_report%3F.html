Por que o relatório Principais arquivos mostra mais hits do que o relatório Downloads?

É comum que o relatório Principais arquivos mostre um número maior de hits do que o relatório Downloads para o mesmo arquivo. Veja o motivo:no caso de um arquivo PDF, muitos navegadores incluem um plug-in que exibe uma página por vez para arquivos PDF.A solicitação de página inicial resulta em um código de status 200 proveniente do servidor da web.Todas as páginas subseqüentes resultam em um código de status 206 (download parcial), já que o plug-in do navegador faz o download apenas da página que está sendo visualizada e não do documento inteiro.Desse modo, no caso de você visualizar cinco páginas PDF do mesmo documento, o servidor da web irá registrar um hit com um código de status 200 e quatro hits com um código de status 206, todos registrados como o mesmo documento PDF.

Veja como os relatórios do Urchin exibem essas informações:

Downloads:um hit de download requer um código de status de 200, 302 ou 304 e, desse modo, o relatório mostrará somente um download para esse arquivo PDF; por exemplo, o hit único com o código de status 200.

O relatório Principais arquivos mostra todas as solicitações de arquivo bem-sucedidas nas quais o código de status é 2XX, 302 ou 304 e nas quais o arquivo corresponde ao tipo MIME correto para ser considerado uma exibição de página.Por padrão, o Urchin trata todos os tipos MIME como exibições de página, exceto no caso de arquivos com estes sufixos:gif, jpg, jpeg, png, js, css, cur, ico, ida.

Desse modo, as cinco solicitações de páginas PDF no exemplo acima seriam contadas como hits para o documento no relatório Principais arquivos, já que cada hit continha um código de status 200 ou 206.

Cada hit de download parcial (status 206) é considerado uma exibição de página distinta, já que cada hit corresponde, mais ou menos, a uma única página em um documento PDF de várias páginas.
