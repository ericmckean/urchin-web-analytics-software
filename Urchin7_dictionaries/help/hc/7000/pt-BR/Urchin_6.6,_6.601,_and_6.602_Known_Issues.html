Problemas conhecidos do Urchin 6.6, 6.601 e do 6.602

Urchin 6.602 - Revisão: 23 de outubro de 2009

Administração

  • O assistente Configuração de meta usa a tecnologia de cobertura do site que não pode lidar com páginas protegidas por senha e é facilmente confundido por páginas complexas, por exemplo, aquelas com frames, javascript, css, redirecionamentos automáticos, etc.

Dados geográficos

  • Devido ao formato do novo banco de dados geográficos, não é mais possível importar informações de mapeamento de nome de IP para domínio.

Processamento de registros

  • Processar registros fora da ordem cronológica, provavelmente, resultará em um processamento de registro muito lento para site de tráfego elevado.
  • Aumentar o limite do banco de dados além do limite padrão de registros padrão de 100,000 pode reduzir o desempenho do processamento de registro, especialmente para sites com grande volume de tráfego.
  • O mecanismo de processamento de registro aloca memória para cada filtro aplicado a um perfil de cada arquivo de registro físico aberto durante o processamento. Isso pode acarretar grandes solicitações de alocação de memória e, possivelmente, falha ao alocar todas as memórias necessárias quando um perfil está configurado com muitos filtros e um grande número de arquivos de registros físicos corresponde à(s) origem(ns) do(s) registro(s), por exemplo, devido a uma correspondência de curinga excessivamente ampla.
  • Grandes tabelas de consulta podem resultar em grande alocação de memória e em possíveis falhas dessa alocação, especialmente, se um grande número de arquivos de registros físicos está aberto durante o processamento de registros.

  • Há alguns problemas conhecidos durante o uso de soquetes UNC. Para obter detalhes, consulte Como trabalhar com origens de registro UNC.

Programadores

  • Programadores do Urchin (de perfil e de CPC) usam o banco de dados para armazenar e recuperar datas e horas a fim de programar vários trabalhos. Isso significa que o banco de dados, uma vez configurado, não deve ter o fuso horário alterado (ou seja, não altere o fuso horário do urchin db em execução na máquina). Além disso, não altere o fuso horário dos programadores do Urchin em execução na máquina.

Comércio eletrônico

  • O Urchin pode ler dados de comércio eletrônico que são incorporados diretamente nos registros de servidor da web quando as transações são registradas via chamadas de UTM javascript ou via um registro de comércio eletrônico ELF2 explícito criado pelo carrinho de compras. O Urchin criará transações duplicadas se o mesmo dado de comércio eletrônico estiver presente nos registros de servidor da web e nos registros ELF2. Sendo assim, é importante que os registros ELF2 não sejam processados se os mesmos dados estiverem incorporados aos registros de servidor da Web.

Utilitários

  • A funcionalidade "dia zero" no udb-sanitizer pode ser muito lenta para perfis com grandes quantidades de dados.

Relatórios

  • Relatórios entre segmentos podem ser muito lentos (ou com limites de tempo completamente fora de questão) para sites com grandes volumes de tráfego e/ou com longos períodos longos.
  • "P/Visitas" resulta diferenças entre o relatório "Conversão média" e o sub-relatório na opção "Opção de análise".

Integração com o Google AdWords

  • O acesso direto ao Google AdWords a partir do Urchin não funcionará se o Urchin 6.6+ estiver instalado na versão de 64-bit do Windows 2008.

Instalação do PostgreSQL

  • Durante a instalação do servidor PostgreSQL em Linux Kernel 2.6 e em Windows, o instalador requer privilégios de root/admin para criar "postgres" de usuário de sistema.
  • Em Linux, o Instalador PostgreSQL requer glibc 2.3.3 ou posterior.
  • Dados de CPC podem não funcionar se o servidor PostgreSQL acompanhado é selecionado durante a instalação do Urchin em Linux Kernel 2.4. Nesse caso, use um servidor PostgreSQL externo.

Urchin 6.601 - Revisão: 23 de julho de 2009

Administração

  • No modo Centro de dados, administradores de conta têm apenas um nível de privilégio, o que é mais restritivo do que os níveis de privilégios de administrador Afiliado encontrados no Urchin 5.
  • O assistente Configuração de meta usa a tecnologia de cobertura do site que não pode lidar com páginas protegidas por senha e é facilmente confundido por páginas complexas, por exemplo, aquelas com frames, javascript, css, redirecionamentos automáticos, etc.

Dados geográficos

  • Devido ao formato do novo banco de dados geográficos, não é mais possível importar informações de mapeamento de nome de IP para domínio.

Processamento de registros

  • Por questões de desempenho, o mecanismo de processamento de registro carrega todo o banco de dados geográfico na memória durante o processamento, resultando em um espaço de memória ainda maior (~800 MB no mínimo), mesmo ao processador perfis de sites muito pequenos. Isso também adiciona uma pequena latência ao tempo de inicialização de processamento de registro para cada perfil, normalmente entre dois e cinco segundos.
  • Processar registros fora da ordem cronológica, provavelmente, resultará em um processamento de registro muito lento para site de tráfego elevado.
  • Aumentar o limite do banco de dados além do limite padrão de 100.000 registros de pode reduzir o desempenho do processamento de registro, especialmente para sites com grandes volumes de tráfego.
  • O mecanismo de processamento de registro aloca memória para cada filtro aplicado a um perfil de cada arquivo de registro físico aberto durante o processamento. Isso pode acarretar grandes solicitações de alocação de memória e, possivelmente, falha ao alocar todas as memórias necessárias quando um perfil está configurado com muitos filtros e um grande número de arquivos de registros físicos corresponde à(s) origem(ns) do(s) registro(s), por exemplo, devido a uma correspondência de curinga excessivamente ampla.
  • Grandes tabelas de consulta podem resultar em grande alocação de memória e em possíveis falhas de alocação de memória, especialmente, se um grande número de arquivos de registros físicos está aberto durante o processamento de registros.

Programadores

  • Programadores do Urchin (de perfil e de CPC) usam o banco de dados para armazenar e recuperar datas e horas a fim de programar vários trabalhos. Isso significa que o banco de dados, uma vez configurado, não deve ter o fuso horário alterado (ou seja, não altere o fuso horário do urchin db em execução na máquina). Além disso, não altere o fuso horário dos programadores do urchin em execução na máquina.

Comércio eletrônico

  • O Urchin pode ler dados de comércio eletrônico que são incorporados diretamente nos registros de servidor da web quando as transações são registradas via chamadas de UTM javascript ou via um registro de comércio eletrônico ELF2 explícito criado pelo carrinho de compras. O Urchin criará transações duplicadas se o mesmo dado de comércio eletrônico estiver presente nos registros de servidor da web e nos registros ELF2. Sendo assim, é importante que os registros ELF2 não sejam processados se os mesmos dados estiverem incorporados aos registros de servidor da Web.

Utilitários

  • Os utilitários de compactação/descompactação fornecidos com o Urchin têm o tamanho limite de 2 GB. Isso pode afetar a capacidade do Urchin de criar backups/arquivos de dados de relatórios para grandes sites ou para usar a opção "compactar" no Destino do registro para origens de registro.
  • A funcionalidade "dia zero" no udb-sanitizer pode ser muito lenta para perfis com grandes quantidades de dados.

Relatórios

  • Relatórios entre segmentos podem ser muito lentos (ou com limites de tempo completamente fora de questão) para sites com grandes volumes de tráfego e/ou longos períodos longos.

Autenticação externa

  • O Urchin 6.6+ vem com módulos de autenticação externos, que incluem LDAP. Atualmente o módulo LDAP não funciona com fornecedores LDAP que sejam executados em "SSL/TLS".

Instalação

  • Atualizar uma instância do Urchin que é usada usando uma licença de Demonstração não é recomendada e pode resultar em problemas de licenciamento/login.

Cookies

  • Para facilitar a interação melhorada na interface de relatórios, o Urchin 6.6+ usa cookies oriundos do cliente. A visualização adequada de relatórios requer que cookies oriundos do cliente sejam ativados no navegador do cliente.

  • Tentar fazer login em uma conta que contenha uma grande quantidade de dados pode resultar no seguinte erro: Erro de sessão ― não foi possível chamar a rotina padrão. (2002) Faça login novamente. Isso foi causado quando a extração de dados para relatórios de totalização excedeu o padrão APPLICATION_TIMEOUT de 45 segundos.
